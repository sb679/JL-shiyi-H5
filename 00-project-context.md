# 项目背景

## 项目名称
JL拾遗 - 校园二手书交易小程序

## 一句话描述
解决本校二手书群信息混乱、书籍难以查找的痛点，提供结构化的书籍发布和浏览平台（信息撮合，不做在线交易）。

## 目标用户
本校学生（预计不超过10万人）

## 核心使用场景
1. **卖书**：学生拍下书籍封面/ISBN → 自动或手动填信息 → 设定价格 → 发布
2. **找书**：学生在书城按分类浏览/搜索关键词 → 看到想要的书 → 表达"想买"意向 → 获取联系方式 → 线下交易
3. **交易完成**：卖家确认售出 → 书籍标记售罄 → 双方互评

## 技术栈与选型理由
| 技术 | 选择理由 |
|------|---------|
| 微信小程序原生框架 | 无需跨端、学习成本低、云开发深度集成、包体积小 |
| 微信云开发（云数据库） | 免服务器运维、免域名备案、免费额度足够 MVP、鉴权内置、`_openid` 自动注入 |
| 云函数（Node.js） | 复杂写操作绕过客户端权限、调用第三方 API、发送通知 |
| OpenLibrary API | 免费、无需 key、无请求次数限制，用于 ISBN → 书籍信息自动补全 |

## 不采用的技术及原因
| 技术 | 排除理由 |
|------|---------|
| 自建后端（Spring/Express 等） | MVP 阶段不需要，增加运维成本和备案复杂度 |
| UniApp / Taro 跨端框架 | 目标仅微信生态，原生更稳定且与云开发集成更好 |
| 云托管（CloudBase Run） | 过度设计，云函数足以满足 v1 需求 |
| 在线支付（微信支付） | 涉及资质审核、手续费、资金结算；本项目定位为信息撮合，非电商平台 |

## 关键制约因素
- 微信小程序备案审核周期不确定
- 微信订阅消息模板需申请，审核可能延迟
- 云开发免费额度有 QPS 限制（20次/秒），高峰需关注
- 无专职设计师，UI 需保持简洁

## 开发周期
v1 MVP 预计 4 周

| 阶段 | 内容 | 预估时间 | 完成状态 |
|------|------|---------|---------|
| 第 1 周 | 项目初始化、登录、数据库建模、发布新书（手动） | 7 天 | ✅ |
| 第 2 周 | 书城列表、搜索、书籍详情页、"我想买"→"确认售出"流程 | 7 天 | ✅ |
| 第 3 周 | ISBN 自动录入、双向评价、"自己的书"页面、订阅消息 | 7 天 | ⚠ 评价/书页已完成，订阅消息延至 v2 |
| 第 4 周 | 测试、修 bug、体验优化、准备提审 | 7 天 | 🔄 当前阶段 |

## 当前版本状态
v1 MVP 功能开发基本完成，进入测试和体验优化阶段。

### 已完成功能
- 云开发自动登录（自动获取 openid）
- 书城首页：在售书籍列表、分类筛选、校区/学院/专业三级筛选、排序、搜索
- 发布新书：ISBN 扫码/手动录入双模式、图片压缩上传、自动关联校区
- 书籍详情页：完整信息展示、多图轮播
- 交易闭环："我想买" → 获取联系方式 → 卖家确认售出
- 交易留言板：买卖双方实时文字交流（售罄后关闭）
- 双向评价：买卖双方互评（匿名展示角色+星级）
- 举报机制：极隐式入口（仅记录，不做自动处理）
- "自己的书"：在售/已售罄双 Tab、下架、数据修复
- 分享到聊天和朋友圈

### 待完善功能
- 用户自定义头像和昵称（微信已废弃自动获取接口）
- 微信订阅消息通知
- 内容安全审核
- 图片/文字上传安全审核
- 管理员举报处理后台

## 成功指标（v1 上线后追踪）
- 日均发布书籍数 ≥ 10 本
- 7 日内售罄率 ≥ 30%
- 用户次日留存 ≥ 40%
