## 2. 云函数部署

### 2.1 部署步骤

1. 在微信开发者工具左侧栏打开"云开发"面板
2. 展开 `cloudfunctions` 目录
3. 右键点击云函数文件夹 → **"上传并部署：云端安装依赖"**
4. 等待部署完成（控制台输出"上传成功"）

**⚠ 关键**：不要使用"上传并部署：所有文件"（跳过安装依赖会导致运行报错）

### 2.2 部署清单（按顺序）

| 顺序 | 云函数 | 备注 |
|------|--------|------|
| 1 | `getOpenid` | 登录必需，最先部署 |
| 2 | `getCampusConfig` | 书城和发布页依赖 |
| 3 | `publishBook` | 核心功能 |
| 4 | `getBookList` | 书城和"自己的书"依赖 |
| 5 | `searchByISBN` | 发布页 ISBN 查询 |
| 6 | `expressInterest` | 详情页"我想买" |
| 7 | `getContact` | 详情页获取联系方式 |
| 8 | `confirmSold` | 详情页确认售出 |
| 9 | `initDatabase` | 数据修复 |
| 10 | `submitEvaluation` | 评价 |
| 11 | `getEvaluations` | 评价列表 |
| 12 | `sendMessage` | 留言板 |
| 13 | `getMessages` | 留言板 |
| 14 | `submitReport` | 举报 |

### 2.3 更新云函数

修改云函数代码后，再次右键 → "上传并部署：云端安装依赖" 即可覆盖。

**⚠ 易错点**：如果修改了 `package.json` 依赖，必须选择"云端安装依赖"，否则新依赖不会安装。

## 3. 数据库索引

在微信开发者工具 → 云开发控制台 → 数据库 → 对应集合 → "索引管理" 中创建。

| 优先级 | 集合 | 索引字段 | 排序 | 用途 |
|-------|------|---------|------|------|
| 必须 | `books` | `category` + `status` + `createTime` | 全部降序 | 书城默认查询 |
| 必须 | `books` | `_openid` + `status` | 降序 | "自己的书"查询 |
| 推荐 | `books` | `category` + `status` + `price` | 升序 | 按价格排序 |
| 可选 | `books` | `isbn` | 升序 | ISBN 查重 |
| 必须 | `evaluations` | `bookId` | 降序 | 按书籍查评价 |
| 必须 | `evaluations` | `toOpenid` + `createTime` | 降序 | 按用户查信用 |
| 推荐 | `messages` | `bookId` + `createTime` | 升序 | 按书籍查留言 |
| 推荐 | `reports` | `targetId` | 降序 | 按目标查举报 |

## 4. 测试指南

### 4.1 单账号测试（开发者工具预览）

**适用场景**：基础 UI 测试、表单校验测试、单页面功能测试

**步骤**：
1. 微信开发者工具点击"编译"（或 Ctrl + B）查看页面效果
2. 可在"模拟器"中切换不同机型测试适配

**局限性**：
- 无法测试涉及两个用户的交互（买家/卖家是同一人）
- 云开发环境在模拟器中可能不可用（需要真机预览）

### 4.2 真机预览（单设备）

**步骤**：
1. 开发者工具点击工具栏 **"预览"** 按钮
2. 扫码在手机微信中打开
3. 测试真实设备上的体验

**适用**：基础功能测试、拍照/扫码功能测试、性能测试

### 4.3 多账号测试（买家 ↔ 卖家）⭐ 推荐

**场景**：测试完整的"发布 → 想买 → 留言 → 确认售出 → 评价"全流程

**方案 A：预览二维码 + 另一个微信号**

```
设备A（开发电脑）：
  → 微信开发者工具登录你的开发者账号
  → 点击"预览" → 生成二维码

设备B（另一台手机）：
  → 用另一个微信号扫码进入
  → 此账号作为"买家"

流程：
  设备A 发布一本书 → 设备B 打开书城 → 看到书 → 点击"我想买"
  → 设备A 切换为卖家 → 看到"想买"列表 → 确认售出
```

**方案 B：体验版**

1. 登录 [微信公众平台](https://mp.weixin.qq.com)
2. "管理" → "成员管理" → "体验成员" → 添加另一个微信号
3. 开发者工具点击"上传" → 填写版本号 → 上传
4. 公众平台 → "版本管理" → 设置为体验版
5. 另一个微信号搜索小程序名即可进入体验版

## 5. 常见问题排查

### 5.1 留言板不显示

**根因**：`getMessages` 云函数未部署或调用失败。

**检查**：
1. 确认 `getMessages` 云函数已上传并部署
2. 确认当前用户是卖家或已点击"我想买"
3. 查看开发者工具 Console 是否有报错

### 5.2 头像/昵称显示默认值

**根因**：微信已废弃 `wx.getUserInfo` 自动获取头像昵称能力。v1 尚未实现用户自定义头像/昵称功能。

### 5.3 云函数调用失败

**检查清单**：
- ✅ 云函数是否已"上传并部署：云端安装依赖"（不是"所有文件"）
- ✅ 云开发环境是否已开通
- ✅ `app.js` 中 `env` 是否配置了正确的环境 ID
- ✅ 开发者工具是否已登录（右上角头像）
- ✅ 真机预览时是否已关联云开发环境

### 5.4 数据库查询无结果

**检查**：
- ✅ 数据库集合是否已创建（`books`, `users`, `evaluations`, `messages`, `reports`, `configs`）
- ✅ 数据库中是否有测试数据
- ✅ 查询条件是否匹配（如筛选分类与数据不一致）

## 7. Git 提交与推送——部署的必要前提

### 7.1 常见误区

| 误区 | 实际情况 |
|------|---------|
| "代码在本地就能部署" | 必须通过 Git 提交推送到远程仓库，CI/CD 和 ECS 才能拉取 |
| "git push 了就能触发 CI" | CI 只在配置指定的分支上触发（如 `main`），feature 分支的 push 不会触发 |
| "GitHub 上能看到文件就是推送成功了" | 嵌套 `.git` 的目录会被当作 submodule（mode 160000），只存指针不存代码 |

### 7.2 推送前检查清单

```powershell
# 1. 确认 Git 有提交历史
git log --oneline -3

# 2. 确认 remote 已配置
git remote -v

# 3. 确认无未提交文件
git status --short

# 4. 确认子目录不是 submodule（应显示 100644 而非 160000）
git ls-files -s jl-shiyi-h5

# 5. 确认在正确分支上推送
git branch

# 6. 推送后验证远程仓库内容
git ls-tree -r main --name-only | findstr "关键文件"
```

### 7.3 嵌套 Git 仓库问题

如果项目目录内有 `.git` 子目录（例如 `jl-shiyi-h5/.git`），外层仓库 `git add` 时会将其识别为 **submodule**，只存储 commit 哈希引用，而非真实文件。GitHub 上看起来文件存在，实际只是一个指针。

**解决方法**：
```powershell
# 1. 移除 submodule 缓存
git rm --cached jl-shiyi-h5

# 2. 删除嵌套 .git 目录
rmdir /s /q jl-shiyi-h5\.git

# 3. 重新添加为真实文件
git add jl-shiyi-h5/

# 4. 提交推送
git commit -m "fix: 解除 submodule 嵌套"
git push origin main
```

**验证修复**：
```powershell
# 应显示 100644（普通文件）而非 160000（submodule）
git ls-files -s jl-shiyi-h5
```

### 7.4 CI/CD 触发条件

GitHub Actions 的触发条件在 `.github/workflows/ci.yml` 中定义：

```yaml
on:
  push:
    branches: [main]  # 只有推送到 main 才会触发
  pull_request:
    branches: [main]  # 指向 main 的 PR 也会触发
```

如果在 `feature/xxx` 分支开发，需要：
1. 合并到 `main` 后推送，或
2. 直接推送到 `main`，或
3. 创建指向 `main` 的 PR
