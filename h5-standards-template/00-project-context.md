# 项目背景

## 项目名称
JL拾遗 - 校园二手书交易 H5 平台

## 一句话描述
面向本校学生的移动优先 H5 二手书信息撮合平台，把群聊中零散、易丢失的二手书信息转为可搜索、可筛选、状态清晰的结构化书城。

## 项目定位
本项目是 H5 网页应用，不再依赖微信小程序、微信云开发、openid、云函数或小程序审核流程。平台只做信息发布、交易意向、联系方式交换、线下成交记录与评价，不做在线支付、担保交易、物流、资金结算。

## 目标用户
- 本校学生，预计总用户不超过 10 万人。
- 主要访问设备为手机浏览器，也应兼容桌面浏览器。
- 用户可能来自微信内置浏览器、手机 Chrome/Safari、学校门户或社群链接。

## 核心使用场景
1. 卖书：学生登录 H5 → 上传封面/背面图片或输入 ISBN → 填写书名、价格、校区、联系方式 → 发布。
2. 找书：学生打开书城 → 搜索书名/作者/ISBN → 按分类、校区、价格筛选 → 查看详情 → 表达想买意向。
3. 沟通：买家表达想买后查看联系方式，也可在站内留言板沟通书况、地点和时间。
4. 成交：买卖双方线下交易，卖家在平台确认售出并选择最终买家。
5. 信任：交易完成后双方互评，后续用户可参考历史评价。

## H5 化后的关键变化
| 原小程序能力 | H5 替代方案 |
|-------------|-------------|
| 微信 openid 自动身份 | 平台账号体系：手机号/邮箱验证码登录，服务端签发 Session 或 JWT |
| 微信云数据库 | PostgreSQL 或 MySQL，推荐 PostgreSQL |
| 云函数 | 独立后端 REST API，推荐 Node.js + NestJS/Express 或等价后端框架 |
| 微信云存储 fileID | 对象存储 URL，使用服务端预签名上传，推荐 S3 兼容存储、阿里 OSS、腾讯 COS 或 Cloudflare R2 |
| wx.scanCode | H5 条形码扫描：优先用摄像头 BarcodeDetector，降级为手动输入 ISBN |
| 微信订阅消息 | 站内消息、邮件通知、短信通知、Web Push 作为后续增强 |
| 小程序分享 | 普通 URL 分享，支持 Open Graph/Twitter Card/微信内预览标题图 |
| 小程序审核 | Web 域名、HTTPS、备案、隐私协议、内容合规审核自管 |

## 推荐技术栈
| 层级 | 推荐选型 | 选择理由 |
|------|----------|----------|
| 前端 | React + Vite + TypeScript | 启动快、生态成熟、适合移动 H5 和后续 PWA |
| 路由 | React Router | 适合多页面业务流，URL 可分享 |
| 状态管理 | TanStack Query + 少量本地 store | 服务端状态、分页、缓存和重试更清晰 |
| UI | 自定义移动端组件或轻量组件库 | 保持校园工具属性，避免营销页风格 |
| 后端 | Node.js + NestJS/Express + TypeScript | 与前端同语言，适合 REST API 和文件上传签名 |
| 数据库 | PostgreSQL | 事务、索引、全文/模糊搜索扩展能力好 |
| ORM | Prisma 或 Drizzle | 类型安全、迁移可追踪 |
| 文件存储 | S3 兼容对象存储/OSS/COS | 图片不进数据库，成本和带宽可控 |
| 缓存 | Redis 可选 | 用于验证码、限流、热点列表缓存 |
| 部署 | Docker + Nginx + HTTPS | H5 静态资源与 API 可独立部署，便于迁移 |

## 不采用的方案及原因
| 方案 | 排除原因 |
|------|----------|
| 纯静态前端无后端 | 无法安全保护联系方式、鉴权、图片上传、确认售出等核心逻辑 |
| 浏览器本地存储全部数据 | 数据无法共享、不可审计、不可检索，不适合交易平台 |
| 在线支付 | 涉及资质、资金风险和纠纷处理，超出信息撮合定位 |
| 完整 IM 系统 | v1 使用轻量交易留言板即可，完整实时聊天后续再做 |
| 过早接入 Elasticsearch | MVP 数据量可先用数据库索引和 ILIKE/trigram，避免运维复杂度 |

## 关键制约因素
- H5 必须有域名和 HTTPS，国内上线通常需要备案。
- 浏览器摄像头能力受权限和浏览器兼容限制，ISBN 扫描必须有手动输入兜底。
- Web 端没有 openid 自动身份，必须设计登录、会话、风控和隐私协议。
- 联系方式属于敏感个人信息，不得在列表、详情初始响应或日志中明文泄露。
- 图片上传需限制大小、格式、数量，并在服务端校验。
- 学生身份真实性取决于登录方案，v1 可不强制校园认证，但要预留校园邮箱/学号认证。

## MVP 边界
### v1 必须包含
- 账号登录与用户资料
- 书城列表、搜索、筛选、排序、分页
- 发布书籍：手动录入 + ISBN 自动补全 + 图片上传
- 书籍详情页
- 表达想买意向
- 鉴权后查看联系方式
- 交易留言板
- 卖家确认售出
- 双向评价
- 举报记录
- 我的发布管理：在售、已售、下架

### v1 不做
- 在线支付、担保交易、退款
- 物流和配送
- 复杂实时 IM
- 管理员完整后台，可先用数据库后台或简易只读页面处理举报
- 强制学生认证，可在 v1.1 增加校园邮箱认证

## 成功指标
| 指标 | 目标 |
|------|------|
| 日均发布书籍数 | ≥ 10 本 |
| 7 日内售罄率 | ≥ 30% |
| 搜索到详情页点击率 | ≥ 20% |
| 用户次日留存 | ≥ 40% |
| 联系方式泄露事故 | 0 |
| P95 列表加载时间 | ≤ 1.5 秒 |
| P95 详情页加载时间 | ≤ 1 秒 |

## 建议开发周期
| 阶段 | 内容 | 预估时间 |
|------|------|---------|
| 第 1 周 | 项目初始化、登录、数据库建模、书籍发布手动版 | 5-7 天 |
| 第 2 周 | 书城列表、搜索筛选、详情、想买和联系方式鉴权 | 5-7 天 |
| 第 3 周 | 图片上传、ISBN 补全、留言板、确认售出 | 5-7 天 |
| 第 4 周 | 评价、举报、测试、部署、隐私协议、体验优化 | 5-7 天 |
